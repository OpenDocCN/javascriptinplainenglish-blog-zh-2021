应该使用TypeScript还是Flow？

TL；博士:选择打字稿。

在为你的JavaScript项目评估一个类型检查器时，你应该选择类型脚本还是流程？

总体而言，您应该考虑:

  • 使用
  • 支持
  • 社区
  • 趋势

使用

类型检查器易于安装和使用吗？有据可查吗？它的性能和速度如何？类型定义多吗？易用性与采用率和生产率直接相关。

支持

维护人员能够支持项目并解决外部贡献吗？

流程:

Flow几乎完全由脸书的一个10人团队开发…该团队的首要任务是满足我们脸书内部客户的需求…遗憾的是，我们没有资源来开发仅由外部开发人员请求的功能。

Flow的代码库非常复杂，很难对Flow本身做出贡献，我们公司内部的构建系统、性能测试和集成测试使得审查和集成此类拉式请求非常耗时，因此我们可能会关闭与我们的优先级不一致的pr。同样，当我们知道我们将无法提供及时的支持时，我们将关闭GitHub问题。

TypeScript有大量的贡献者，在 GitHub 上非常活跃。

社区

社区很好地支持类型检查器吗？其他开发者喜欢用吗？你能在谷歌或 Stackoverflow 上轻松找到答案吗？问题和拉动请求是否得到充分解决？其他开发人员会为类型检查器构建工具、插件和集成吗？一个强大的社区等同于一个稳固的行业地位。

趋势

类型检查器的未来是什么样子的？贡献者在发布功能和修复bug吗？公司和学校是否会选择其中一种类型的检验员？这可以确定拥有类型检查经验的开发人员的数量。问问你自己，“五年后，哪个类型检查器会赢？”跟随错误的趋势，将会增加技术债务和过时的技能，这些在以后会成为问题。

结论

您应该选择TypeScript，因为它具有:

  • 更好的性能
  • 大力支持
  • 一个大社区
  • 更多类型定义(见明确类型化
  • 更好的 Visual Studio代码集成
  • 更多动力(见谷歌趋势和 npm-stat

此外，《心流》的创作者脸书正在将他们的几个大项目如 Yarn 和 Jest 迁移到打字稿。当心流的创造者离开心流时，这是一个危险信号。
