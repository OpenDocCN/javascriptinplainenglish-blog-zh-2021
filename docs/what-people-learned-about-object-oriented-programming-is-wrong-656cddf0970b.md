OOP 是一种范式，它给我们带来了一系列可能性，让我们可以从更优雅、更优化的代码中创建更高效的软件。然而，要从这些潜力中获益，有必要真正理解什么是面向对象。

许多大学教授的 OOP 与现实脱节。因此，这个编程模型的关键部分被省略了。因此，很多时间都花在教授没有实际应用的例子上，这些例子无助于解决现实生活中的问题。

在本文中，我将向您展示为什么许多人学习 OOP 是错误的，以及后果。我还将谈到面向对象的真正资产，这在大学课堂上经常被忽略。

# **面向类还是面向对象？(什么不是 OOP)**

例如，一个主要的问题是，大多数人实际上学会了面向类而不是面向对象。

我记得，在大学里，花一整个学期的时间去研究没有任何意义的抽象图表和动物等级的例子，这种折磨仿佛就发生在昨天。当焦点放在类建模上时，所有对象间交互的真正力量都被忽略了。

这是折磨，不仅仅是折磨。然而，这是一种浪费，因为面向对象的本质能够在简单优雅的代码中具体化复杂的思想、算法和抽象。

我的感觉是，它侧重于结构，而不是过程。结果，人们开始考虑静态对象，在程序执行期间不与对象交互。这种线性和顺序的结构反映了更多的过程化编程，它与步骤一起工作，而不是 OOP 带来的符号表示的能力。

正如我已经提到的，在 OOP 中，知道如何创建类、定义方法、属性的公共或私有访问模式是不够的。您需要很好地协调对象之间的关系。为此，有必要使用抽象的力量来思考我们想要的代码编写结果，而不要仅仅停留在任务上。

为了强化这种心态，有必要理解 OOP 的巨大资产，它通常没有得到必要的重视。这是我们下一个话题要讲的。

# **被遗忘的宝藏**

OOP 允许我们用过程的不断改进来创建软件，而这并不意味着整个系统的重新安排。你不需要检查整个代码并重新培训每个人，因为你改变了整个过程，没有什么能再工作了。这是过程焦点和面向对象焦点之间的区别。

OOP 给了你一个更好的接口，这样就有了合作，而不需要规范对方的行为。这正是一个好团队的特质。一个好的团队，在一起工作时，不会专注于任务。相反，它关注结果。这是信任的典范，而不是控制。

# **这两种范式之间差异的真实例子**

有一次，我不得不去银行解决一所大学和他们的财务总监的问题，令人惊讶的是，少了一名服务员，他离开柜台去控制寻求援助的队列。他按照到达的顺序写下了人们的名字和 CPF。因此，他在观察其他服务员，当一个服务员被释放时，他叫这个人的名字，把他们转到正确的柜台，并通过划掉这个人的名字来更新他们的控制。

这显然是一个程序性的例子，银行分配资源来控制整个过程，累积了两个责任:参加和管理队列。

他不再专注于尽可能提供最好的服务，而是降低了他执行核心活动的能力，将精力花在一项中间活动上。这种浪费无疑增加了顾客排队的处理时间。

每当您的代码中有一个负责选择下一项的“for”或“while”循环与处理该项本身相混合时，就会看到这个例子。一对经典夫妇。

# **有效性范例**

另一方面，前几天，我必须带我的朋友去打疫苗，健康中心实施了不同的策略。护士们完全专注于集中她们为数不多的资源，尽可能地为人们提供最好的护理。

他们没有办法组织一个队列，并把这个责任委托给系统的另一部分。

当我和我的朋友到达时，我看到人们在门前等着，就问:谁是最后一个排队的？他们给我指了指那个人，我确认了一下，介绍自己是新来的最后一个，然后开始等待。不久后，另一个人来了，和我玩同样的动态，也在等待。

然后门开了，护士喊道:下一个。然后第一个排队的人站起来走向服务。从那以后，每次门一开，护士就尖叫着那个神奇的词:下一个。

它展示了队列组织和服务本身之间的职责的完全分离。对于护士来说，病人被封装在一个迭代器对象中，每当他们接收到消息“close”时，只提供下一个病人，而不管他们是谁以及他们是如何组织的。

在这种情况下，在迭代器中，患者被组织成一个双向链表，其中给定的患者只知道他的前任和他的继任者，但不必担心整个列表的顺序。

这种职责分离甚至允许队列改变，而完全不影响护士的程序。因为对他们来说，最重要的是知道谁是下一个。您不需要知道用于组织事物的名称、索引或内部数据结构——只有下一个才重要。

这是你的力量，让你远离对控制和信任的关注，用更好定义的接口让对象合作解决问题。
